# 销售蓝表 — 实现方案

> 基于 Miller Heiman 策略销售方法论，结合崔建中"价值销售"体系适配
>
> 最后更新：2026-07-06（第2版，补充边界情况与交互细节）

---

## 一、概述

### 1.1 什么是销售蓝表

销售蓝表是一张结构化的大客户销售分析表，用于替代当前销售机会中纯主观的赢率输入。通过多维度评估，将赢率从一个"拍脑袋的数字"转变为"有依据的评分"。

### 1.2 使用规则

| 阶段 | 蓝表要求 | 赢率列显示 |
|------|---------|-----------|
| **信息** | 无需填写 | 不显示赢率列 |
| **线索** | **必须填写**后才能晋升到"机会" | 未填显示 `—`，已填显示计算值 |
| **机会/投标/议价** | 可更新，建议每次阶段变更时重新评估 | 默认显示蓝表计算值，可微调 |

## 二、蓝表弹窗界面设计

### 2.1 弹出方式

点击机会列表中的**赢率单元格/Popover**，弹出蓝表 Modal（尺寸约 640×480，可滚动）。

### 2.2 界面布局

蓝表分三个 Tab：

#### Tab 1：否决项

两个否决项，各自独立判断：

```
┌──────────────────────────────────────────────┐
│  ⛔ 否决检查                                  │
│                                              │
│  预算匹配度                                   │
│  ○ 预算充足  ○ 预算可争取  ● 预算严重不足 ❌   │
│                                              │
│  时间匹配度                                   │
│  ○ 时间充裕  ● 时间紧凑可行  ○ 时间无法满足 ❌  │
│                                              │
│  任一选择"❌" → 整单否决，赢率 0%             │
└──────────────────────────────────────────────┘
```

**规则：**
- 预算/时间各自独立选择
- 任一选择"❌"时，弹出确认提示，返回后赢率锁定 0%，其他 Tab 不可编辑
- 两个都正常（无 ❌）时，进入 Tab 2

#### Tab 2：采购角色矩阵

```
┌─────────────────────────────────────────────────────────────────────┐
│  👥 采购角色矩阵                             [+ 添加角色]           │
│                                                                     │
│  ┌────────┬────────┬────────┬────────┬────────┬────────┬──────────┐│
│  │ 角色   │ 姓名   │ 影响力  │ 支持度  │ 需求   │ 竞争   │ 客户    ││
│  │        │ (选填) │ (高/中/低)│(-5~+5)│ 匹配度  │ 定位   │ 关系    ││
│  ├────────┼────────┼────────┼────────┼────────┼────────┼──────────┤│
│  │ EB     │        │ ▾ 高   │  +3    │  4     │  3    │  3       ││
│  │ UB     │        │ ▾ 中   │  +2    │  3     │  2    │  2       ││
│  │ TB     │        │ ▾ 低   │  -1    │  2     │  2    │  1       ││
│  │ Coach  │        │ ▾ 中   │  +4    │  4     │  4    │  4       ││
│  └────────┴────────┴────────┴────────┴────────┴────────┴──────────┘│
│                                                                     │
│  默认角色：EB / UB / TB / Coach                                     │
│  可点击「+ 添加角色」新增自定义角色                                │
└─────────────────────────────────────────────────────────────────────┘
```

**各字段说明：**

| 字段 | 输入方式 | 说明 |
|------|---------|------|
| **角色** | 固定四项 + 可自定义 | EB/UB/TB/Coach 为固定角色 |
| **姓名** | 文本输入 | 选填，方便关联联系人 |
| **影响力** | 下拉：高/中/低 | 该角色在决策中的话语权 |
| **支持度** | 下拉或滑条：-5 ~ +5 | **核心指标**，默认 -1（未知态度） |
| **需求匹配度** | 评分 1~5 | 我方方案满足该角色需求的程度 |
| **竞争定位** | 评分 1~5 | 对比竞对我方在该角色眼中的优势 |
| **客户关系** | 评分 1~5 | 我们与该角色的关系质量 |

**支持度评分标准：**

| 分数 | 立场 | 典型表现 |
|------|------|---------|
| +5 | 热情拥护 | 主动帮你推动项目、提供内部信息 |
| +4 | 强烈支持 | 积极表态支持，愿意引荐关键人 |
| +3 | 支持 | 明确表示支持，给予有效信息 |
| +2 | 感兴趣 | 有兴趣了解，愿意听方案 |
| +1 | 略有好感 | 初步正面印象 |
| 0 | 中立 | 态度模糊（**实际极少**） |
| -1 | **默认值** | 轻微负面倾向或未明确表态 |
| -2 | 不认可 | 持怀疑态度 |
| -3 | 反对 | 明确反对 |
| -4 | 强烈反对 | 积极反对，表达不满 |
| -5 | 敌对 | 全力阻止项目推进 |

> **重要规则**：不知道对方态度的，默认为 **-1**，不能视为 0。
>
> **UI 实现要求**：新增角色行时，支持度下拉/滑条**默认选中 -1**，而非 0。
>
> **角色表为空处理**：若角色表无任何行（`roles.length === 0`），`Σwi = 0` 会导致除零错误。此时赢率直接返回 0%，蓝表弹窗中显示提示“请至少添加一个采购角色”。

#### Tab 3：策略计划

```

│                                              │
│  反应模式                                     │
│  ○ G(增长)  ○ T(困难)  ● EK(平稳)  ○ OC(自满) │
│                                              │
│  下一步行动                                   │
│  ┌─────────────────────────────────────┐     │
│  │ 1. 约见 EB 张总，了解预算审批流程      │     │
│  │ 2. 安排技术交流会打消 TB 顾虑          │     │
│  │ 3. ...                              │     │
│  └─────────────────────────────────────┘     │
│                                              │
│  目标角色                                     │
│  ☑ EB（张总）→ 目标支持度从 +2 提升到 +4      │
│  ☑ TB（李工）→ 目标支持度从 -1 提升到 +2      │
└──────────────────────────────────────────────┘
```

**10级定位标尺（参考值）：**

| 定位 | 含义 |
|------|------|
| 陶醉 | 客户极度认可，基本拿下 |
| 太棒了 | 客户非常满意 |
| 安稳 | 项目稳固，无明显风险 |
| 舒服 | 进展顺利，关系良好 |
| OK | 正常推进 |
| 顾虑 | 客户有未被解决的顾虑 |
| 不舒服 | 出现明显问题 |
| 担心 | 有竞争对手领先或我方劣势 |
| 害怕 | 项目可能丢失 |
| 惊慌 | 基本确定要输了 |

**反应模式：**

| 模式 | 含义 | 策略 |
|------|------|------|
| G（Grow） | 想变得更好 | 容易争取支持，但有竞争 |
| T（Trouble） | 遇到麻烦 | 支持意愿强，抓紧解决核心问题 |
| EK（Even Keel） | 现状挺好 | 难获得支持，需制造痛点 |
| OC（Over Confident） | 过度自满 | 极难，除非客户自身觉醒 |

## 三、赢率计算方法

### 3.1 计算流程

```
输入：角色矩阵数据（支持度、影响力、各维度分）
输出：建议赢率（0~100%）

步骤 1 ── 否决检查
  预算或时间任一"❌" → 赢率 = 0%，终止

步骤 2 ── 角色缺失修正
  未识别 EB（角色表中无 EB 行）→ × 0.5
  未识别 Coach              → × 0.85
  两者都未识别               → × 0.4

步骤 3 ── 基础支持度分
  影响力权重映射：
    高 → 5，中 → 3，低 → 1

  支持度归一化：(支持度 + 5) / 10（-5→0%, 0→50%, +5→100%）

  加权平均 = Σ(各角色影响力权重 × 支持度归一化) / Σ(影响力权重)

  基础分 = 加权平均 × 100

步骤 4 ── 关系放大器
  关系加权 = Σ(各角色影响力权重 × 客户关系评分) / Σ(影响力权重)
  关系因子 = 1 + (关系加权 - 2.5) × 0.05
  关系因子范围：0.85 ~ 1.15

步骤 5 ── 反应模式修正
  OC（过度自满）× 0.8
  其他 × 1.0

步骤 6 ── 最终计算
  赢率 = 基础分 × 角色缺失修正 × 关系因子 × 反应模式修正
  最终赢率 = Math.min(Math.round(赢率), 100)
```

### 3.2 完整计算公式

```
WinRate = SupportScore × RolePenalty × RelationshipFactor × ReactionFactor

其中：
  SupportScore = Σ(wi × (si + 5) / 10) / Σ(wi) × 100
    wi = 角色影响力权重（高=5, 中=3, 低=1）
    si = 支持度评分（-5 ~ +5）

  RolePenalty =
    无 EB 且 无 Coach → 0.4
    无 EB → 0.5
    无 Coach → 0.85
    全部识别 → 1.0

  RelationshipFactor = 1 + (avgRel - 2.5) × 0.05
    avgRel = Σ(wi × rel_i) / Σ(wi)
    rel_i = 客户关系评分（1~5）
    结果截取范围：0.85 ~ 1.15

  ReactionFactor =
    OC → 0.8
    其他 → 1.0

  最终：WinRate = Math.min(Math.round(WinRate), 100)
```

### 3.3 示例计算

**项目 A：**
| 角色 | 影响力 | 支持度 | 关系 |
|------|--------|-------|------|
| EB | 高(5) | +3(80%) | 3 |
| UB | 中(3) | +2(70%) | 2 |
| TB | 中(3) | -1(40%) | 2 |
| Coach | 中(3) | +4(90%) | 4 |

```
支持加权 = (5×80% + 3×70% + 3×40% + 3×90%) / (5+3+3+3)
         = (4.0 + 2.1 + 1.2 + 2.7) / 14 = 10.0/14 = 71.4%
基础分 = 71.4

角色缺失：全部识别 → × 1.0

关系加权 = (5×3 + 3×2 + 3×2 + 3×4) / 14 = (15+6+6+12)/14 = 39/14 = 2.79
关系因子 = 1 + (2.79 - 2.5) × 0.05 = 1.014

反应模式 EK → × 1.0

WinRate = 71.4 × 1.0 × 1.014 × 1.0 = 72.4 → 72%
```

## 四、数据结构

### 4.1 新增 TypeScript 类型

```typescript
// 蓝表角色条目
interface BlueTableRole {
  id: string;
  roleType: 'EB' | 'UB' | 'TB' | 'COACH' | 'CUSTOM';
  roleLabel?: string;        // 自定义角色名称
  name?: string;             // 联系人姓名
  influence: 'high' | 'medium' | 'low';  // 影响力
  support: number;           // 支持度 -5 ~ +5
  demandFit: number;         // 需求匹配度 1~5
  competition: number;       // 竞争定位 1~5
  relationship: number;      // 客户关系 1~5
}

// 蓝表完整数据
interface BlueTable {
  vitoCheck: {                // 否决项
    budget: 'ok' | 'possible' | 'failed';
    timeline: 'ok' | 'possible' | 'failed';
  };
  roles: BlueTableRole[];
  positioning: number;        // 定位 1~10
  reactionMode: 'G' | 'T' | 'EK' | 'OC';
  strategy: string;           // 下一步行动文本
  targets: {                  // 目标角色
    roleId: string;
    targetSupport: number;    // 目标支持度
  }[];
  calculatedWinRate: number;  // 自动计算值
  adjustedWinRate?: number;   // 手动微调值
  updatedAt: string;
}
```

### 4.2 机会数据类型扩展

```typescript
// 在 SalesOpportunity 中增加字段
interface SalesOpportunity {
  // ... 现有字段
  blueTable?: BlueTable;     // 蓝表数据（可选）

}
```

## 五、UI 交互说明

### 5.1 赢率列变化

| 阶段 | 无蓝表 | 已填蓝表 |
|------|--------|---------|
| **信息** | 不显示该列 | — |
| **线索** | 显示 `—`（灰色） | 显示计算值，可点击重新编辑 |
| **机会/投标/议价** | 显示 `0%`（灰色） | 显示计算值，可点击编辑 |
| **中标** | 固定 100% | — |

### 5.2 阶段晋升校验

从**线索晋升到机会**时：
- 检查是否存在 `blueTable` 且 `roles.length > 0`
- 未填写则弹窗提示"请先填写销售蓝表"，阻止晋升

### 5.3 蓝表完整度指标

蓝表弹窗底部显示完整度评分（不参与赢率计算），帮助用户判断填写质量：

```
📊 蓝表完整度 4/5
✅ 已识别 EB
✅ 已有 Coach
✅ 所有角色支持度已填写
✅ 策略计划已填写
⬜ 目标角色未设定
```

### 5.4 赢率手动微调

蓝表自动计算赢率后，用户可在蓝表底部手动微调（±10% 范围），用于覆盖特殊情况。微调后的值带标记，系统记录原始计算值。

**微调值重置规则**：当用户重新编辑蓝表（修改任何角色数据或否决项）后，之前的手动微调值**自动清除**，恢复为自动计算值。

### 5.5 清空蓝表

蓝表弹窗底部提供「清空蓝表」按钮（带二次确认）：
- 清空后 `blueTable` 设为 `undefined`
- 赢率恢复为 0%
- 线索阶段恢复显示 `—`
