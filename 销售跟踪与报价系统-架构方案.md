# 销售管理与报价系统 — 架构方案

> 最后更新：2026-06-26

---

## 一、角色与权限

| 角色 | 职责 | 核心权限 |
|---|---|---|
| **销售经理** | 销售管理、销售支持、组织报价与跟进 | 管理所有销售机会、创建报价单、推动销售流程、生成报价表 |
| **方案经理** | 维护组件/物料数据库、编制技术方案和概算表 | 物料CRUD、编制/编辑概算表（所有版本）、上传项目方案截图 |
| **交付经理** | 赢单后接手项目交付 | 更新交付节点进度、确认验收回传、录入实际成本（按概算表结构） |
| **总监** | 审核概算表、审核物料变更、系统管理 | 审核所有版本的概算表、审核物料CRUD、用户管理/基础配置 |

> 注：区域销售没有系统账号，其信息由销售经理统一录入。总监同时担任系统管理员角色。

## 四、项目状态流转

```
                      ┌──────────┐
                      │  过程中   │◄──── 新建/编辑中
                      └────┬─────┘
                           │
             ┌─────────────┼──────────────┐
             ▼             ▼              ▼
         ┌──────┐     ┌──────┐      ┌──────┐
         │  赢  │     │  输  │      │ 冻结 │
         └──────┘     └──┬───┘      └──┬───┘
                         │             │
                    ┌────┴────┬───────┐ │ 可恢复为过程中
                    ▼         ▼       ▼ │
                 ┌──────┐ ┌──────┐ ┌──────┐
                 │ 竞对  │ │ 取消 │ │ 放弃 │
                 └──────┘ └──────┘ └──────┘
```

**状态变更限制：**
- **信息/线索阶段**：不可赢单（防止漏报或捡单），只能切换到输或冻结
- **过程中**：可切换到赢/输/冻结
- **冻结**：可恢复为过程中

**输单原因三大类：** 竞对（对手优势导致丢单）/ 取消（客户方取消）/ 放弃（我方主动退出）

**赢单需选择原因**（竞对正面因素：价格优势/技术方案/客户关系/品牌）

**状态变更弹窗**：层级多选 → 先选大类（输时），再选子因，同大类可多选

**竞争对手列规则：**
- 多个竞争对手用 `、` 或 `，` 分隔
- 输单时如果竞争对手 > 1 家，弹窗中需选择最终赢家
- 确认输单后 competitor 字段自动更新为所选赢家
- 使用非法分隔符（如 `,` `/`）时单元格右侧显示红色 `!` 警告

## 七、交付管理

### 交付节点生命周期（15个节点）

```
赢单 → 交付经理接手
  → Handover → 合同签订 → Kickoff → 方案细化 → 技术会签
  → 详细设计 → 设计评审 → 制造采购 → 组装调试 → 出厂验收
  → 包装发货 → 现场安调 → 验收整改 → 终验收 → 项目总结 → 关闭
```

| 序号 | 节点 | 说明 |
|---|---|---|
| 1 | Handover | 项目交接（销售→交付） |
| 2 | 合同签订 | 正式合同签署 |
| 3 | Kickoff | 项目启动会 |
| 4 | 方案细化 | 详细方案设计 |
| 5 | 技术会签 | 客户确认技术方案 |
| 6 | 详细设计 | 出图/编程 |
| 7 | 设计评审 | 内部评审 |
| 8 | 制造采购 | 物料采购 + 生产制造 |
| 9 | 组装调试 | 厂内组装和预调试 |
| 10 | 出厂验收 | 客户到厂预验收 |
| 11 | 包装发货 | 设备包装发运 |
| 12 | 现场安调 | 现场安装调试 |
| 13 | 验收整改 | 验收问题整改 |
| 14 | 终验收 | 客户最终验收签字 |
| 15 | 项目总结 | 项目复盘、成本核算 |

### 节点管理机制

交付经理负责维护每个节点的：
- **状态**：未开始 / 进行中 / 已完成 / 已延期（点击循环切换）
- **计划日期**：可设定和修改，每次修改自动记录变更历史
- **实际日期**：状态改为"已完成"时，系统自动填入当前日期
- **历史记录**：仅记录计划日期修改历史，节点状态变更不记历史

### 实施计划审批流

```
交付经理设定所有节点的计划时间
  → 点击"提交实施计划"
  → 状态变为"待审批"
  → 总监审批
      ├── 通过 → 所有节点计划时间锁定（可修改，但修改后系统记录变更历史）
      └── 驳回 → 退回交付经理修改后重新提交
```

- 审批通过后，交付经理如需修改计划时间，可以修改，系统记录修改历史
- 计划锁定后，日期显示为只读文本

### 成本对比审批流

成本对比基于**报价表的逐条项次**（GroupItem），不是按节点分配成本。

```
交付经理在各报价项次中输入实际成本
  → 点击"提交成本对比"
  → 状态变为"待审批"
  → 总监审批
      ├── 通过 → 成本数据锁定（只读），作为交付经理和方案经理的考核依据
      └── 驳回 → 退回修改后重新提交
```

### 实际成本对比规则

| 对比维度 | 差异红线 | 说明 |
|---|---|---|
| **分项成本** | **-5% ~ +10%** | 超出此范围标红显示"超标" |
| **总成本** | **-2.5% ~ +5%** | 超出此范围标红显示"超出正常范围" |

- 只比较**成本**，不比较售价
- 偏差项次标红，正常显示绿色
- 仅在成本对比审批通过后数据锁定，作为考核依据

### 项目完成条件

同时满足以下两个条件时，可点击「完成项目」按钮将项目标记为已完成：
1. **节点15（项目总结）状态为"已完成"**
2. **成本对比审批通过**

已完成的项目自动从交付管理首页的"进行中"列表移至"已完成"列表。

### 交付管理页面结构

- **首页**（`/delivery`）：列表化展示，分为"进行中"和"已完成"两个标签
  - 每个项目卡片显示：进度圆环、客户名、项目名、合同金额、计划/成本审批状态
  - 点击卡片进入详情页
- **详情页**（`/delivery/:id`）：两个 Segmented 标签
  - **实施计划**：15节点时间轴 + 审批状态条
  - **成本对比**：报价表项次逐条对比 + 审批状态条

## 八、审核工作流

### 概算表审核（所有版本都需要）

**版本生命周期：**
```
方案经理编制（草稿状态，可见但不可生成报价单）
        │
        ▼ 手动点击"提交审核"
方案经理 提交审核 ──► 总监 审核
                          ├── 通过 → 版本生效，可生成报价表
                          └── 驳回 → 退回方案经理修改后重新提交
```

**特殊规则：**
- 每次点击"提交审核"生成一个新版本号，中间修改保存不递增版本号
- 方案经理的报价编制页面**不显示生成报价表按钮**
- 销售经理可以查看所有版本的概算表，但总监未审批或审批不通过的版本**不可生成报价表**
- **毛利率 GP3 < 15%** 时，必须总监审批通过才能生成报价表
- 未审核通过或草稿状态的版本：**可见但不可生成报价表**

### 物料/组件数据库审核
```
方案经理 提交物料变更 ──► 总监 审核
                                    ├── 通过 → 物料生效
                                    └── 驳回 → 退回修改
```

### 交付管理两级审批
1. **实施计划审批**：交付经理提交节点计划时间 → 总监审批 → 通过后计划时间锁定
2. **成本对比审批**：交付经理提交报价项次实际成本 → 总监审批 → 通过后成本数据锁定，作为考核依据

## 九、数据分析模块

### 9.0 统一原因分类体系（2026-06-26 建立）

所有状态变更的原因统一归入 `src/reasonTaxonomy.ts`，通过 `parseReasons()` / `formatReasons()` 序列化/反序列化。

**存储格式：** `竞对:价格:主机价格,解决方案价格;竞对:客户关系`
- `:` 分隔层级（大类:子类:具体项）
- `;` 分隔多选
- `,` 分隔同一子类下的多个具体项

**赢单原因（正向表述）：**
```
竞对: 价格(主机成本/解决方案成本), 技术方案(痛点发掘和解决/主机性能), 客户关系, 品牌
```

**输单原因：**
```
竞对: 价格(主机价格/解决方案价格), 技术方案(痛点发掘和解决/主机性能), 客户关系, 品牌
取消: 预算缩减, 投资延后, 需求变更, 市场变化
放弃: 利润率过低, 风险过高, 技术不达标, 交付能力不足, 介入时间晚
```

**冻结原因：** 预算缩减, 投资延后, 需求变更, 市场变化

### 9.2 赢单分析

**赢单原因柱状图：** 6 个叶子子因（主机成本/解决方案成本/痛点发掘和解决/主机性能/客户关系/品牌）各自显示次数，含零值显示 `—`，柱间距可调。

**击败竞争对手：** 赢单卡片底部显示打败的竞争对手列表（红框横向条），最多显示 5 个，数据来自机会的 `competitor` 字段（支持 `、` 分隔多值）。

### 9.3 输单分析

**三原因横向堆叠图（已实现）：**
- Y 轴：原因大类名称（竞对/取消/放弃），左侧 64px 固定宽度
- X 轴：次数，等距网格线，底部标签 `0 1 2 ... maxCount`
- 子因分段：蓝/灰实线交替描边
- 坐标系：Y 轴标签 + 网格竖线 + X 轴标签

**竞争对手输单柱状图（已实现）：**
- 竖状柱状图，显示各竞争对手导致输单的项目数（前 10 名）
- 数据来自 `competitor` 字段，按 `、`/`，` 拆分

### 9.4 冻结分析
- 冻结项目数量/金额
- 冻结原因（预算缩减/投资延后/需求变更/市场变化）

### 9.5 销售漏斗分析
- 四阶段 SVG 漏斗（信息→线索→机会→中标）
- 各阶段数量/金额标注（当期 + 财年累计）
- 阶段间转化率（按数量/金额双维度，漏斗内居中显示）
- 漏斗填充 `rgba(0,80,158,0.12)`，轮廓线 2.5px，分割线虚线 1.5px
- 右侧引出线 + 端点圆点标注

### 9.6 销售排行（4×2 竖状柱状图网格）

| 维度 | 计算方式 | 单位 |
|---|---|---|
| 销售额 | Σ(已赢单金额) | K |
| 利润额 | Σ(赢单金额×报价利润率) | K |
| 管道潜力 | Σ(活跃项目金额×赢率%)，排除赢单 | K |
| 转化效率 | 成交金额÷总标金额(含输/冻结) | % |
| 项目单价 | 总金额÷项目数 | K |
| 输单金额 | Σ(输单金额) | K |
| 漏斗健康度 | 线索数+机会/投标/议价数（堆叠） | 个 |

**柱状图规范：** SVG 竖状，不填充仅描边（strokeWidth=3），蓝/紫两色交替，直角，固定 10 卡位，Y 轴 4 等分网格线 + 均值虚线（橙色）。
